# 平台基座中文设计稿

状态：draft
范围：控制面后端、控制面前端、根级治理、后续 agent worker
更新时间：2026-05-20

## 1. 设计目标

把当前控制面升级为一个可长期维护的 AI 开发平台基座。它不是任务列表的重新包装，也不是在现有页面上继续补字段，而是一个能承载需求拆解、并行 agent 调度、质量门禁、自愈、发布验收和项目体检的工作台。

操作员的默认角色是查看、判断和批准，不是执行命令、修服务、看日志或手工发布。

当前真实能力评级见：`docs/contracts/PLATFORM_CAPABILITY_MATRIX_CN.md`；机器可读矩阵见：`docs/contracts/platform-capability-matrix.json`。

Recovery Engine 重设计合同见：`docs/contracts/RECOVERY_ENGINE_REDESIGN_CN.md`。当前 watchdog 不能视为恢复引擎底座，只能作为历史能力和反例输入。

## 2. 默认响应式要求

后续所有包含前端的项目，除非任务或项目规则明确排除，都必须同时设计和开发：

- PC Web：默认主工作台，适合高密度信息、表格、任务流、调度和审计。
- 手机尺寸：独立信息架构，适合状态浏览、待决策处理、风险确认和轻量操作。

手机页面不能默认视为 PC 页面压缩版。需要根据场景重新组织信息密度、导航层级、操作入口和关键任务。

## 3. 产品信息架构

### PC Web 工作台

PC 端服务于“全局掌控”和“深入排查”。

主要区域：

- 左侧导航：总览、项目、任务流、Agents、风险、治理。
- 顶部状态条：平台健康、API 延迟、错误率、可用性、活跃 Agents、队列状态、更新时间。
- 总览指标：项目总数、进行中项目、任务总数、进行中任务、已发布服务、需人工决策。
- 活跃工作：项目、阶段、当前任务、负责人 Agent、进度、更新时间。
- 任务流：需求、拆解、子任务、Review、发布、Live 验证的横向生命周期。
- Agents 池：agent、角色、模型、状态、当前负载。
- 能力成熟度：能力域、成熟度、趋势。
- 风险队列：风险、严重度、证据、状态、发现时间。
- 需人工决策：决策事项、上下文、建议、影响范围、发起 agent、等待时长。

### 手机工作台

手机端服务于“快速判断”和“轻量处理”。

主要区域：

- 顶部摘要：平台健康、阻塞数、待决策数。
- 今日关注：最重要的运行态、风险和待决策。
- 任务流快照：以纵向步骤展示关键任务节点。
- Agent 状态：只展示异常、忙碌和需要关注的 agent。
- 风险与决策：合并成一个优先级列表，提供查看、处理、延期。
- 底部导航：总览、任务、风险、Agents、治理。

手机端原则：

- 首屏只回答“现在是否健康、我是否需要介入”。
- 次级详情采用下钻页面或抽屉，不在首屏塞宽表。
- 关键操作使用明确按钮，不依赖 hover。
- 长表格改成分组列表，但保留证据和状态。

## 4. 视觉方向

目标气质：成熟、克制、企业级、工程可信。

颜色：

- 背景：冷白和浅灰，不使用米色、奶油色或大面积深蓝。
- 主色：蓝色，表达执行和可操作。
- 辅色：青绿色，表达健康和验证。
- 风险色：琥珀、红色，仅用于需要关注的异常。
- 中性色：深墨文字、灰蓝边框、浅灰分隔。

排版：

- 标题短、清晰、偏工具型。
- 数字指标使用较大字号，但不做营销化夸张。
- 表格、标签、状态 chip 保持高可读密度。

组件：

- 8px 左右圆角。
- 少量轻阴影，主要靠边框和留白分层。
- 避免嵌套卡片。
- 任务流用轨道和节点，不用大块插画。
- 风险与成熟度用紧凑 chip，不用彩色大卡。

## 5. 能力成熟度设计

每项平台能力必须明确成熟度，避免把雏形误读为完整能力。

| 成熟度 | 含义 | UI 文案 |
| --- | --- | --- |
| `none` | 还没有实现 | 未接入 |
| `scaffold` | 有数据形态或占位，但不可依赖 | 设计中 |
| `partial` | 部分路径可用，缺契约、恢复或可观测性 | 部分可用 |
| `usable` | 有契约、测试和中台可见反馈 | 可用 |
| `production` | 有广覆盖、恢复、告警和真实运行证明 | 生产级 |

首批能力域：

- 工作台总览
- 任务 DAG
- 调度锁域
- 模型路由
- Agent 池
- 事件源状态
- Durable artifacts
- Recovery Engine
- Workflow gates
- CI/CD 门禁
- LLM Reviewer
- 定期项目体检
- 快速定位 skill

## 6. 任务流设计

标准生命周期：

```text
需求 -> 拆解 -> 子任务 -> Review -> 发布 -> Live 验证 -> 验收
```

每个节点需要显示：

- 状态：待开始、进行中、通过、阻塞、失败。
- 证据：artifact、测试报告、review 报告、发布 manifest、浏览器验收。
- 负责人：agent 或系统模块。
- 锁域：当前占用的项目、路径、runtime、publish、DB、edge。

## 7. Agent 池设计

Agent 不只显示是否在线，还要显示它在平台里的角色。

角色：

- planner：需求拆解、架构、风险。
- implementer：实现、测试、修复。
- locator：快速定位、日志摘要、代码地图。
- reviewer：质量门禁、设计审查。
- recovery：自动恢复、阻塞修复。

展示字段：

- Agent ID
- 角色
- 模型
- 状态
- 当前负载
- 锁域
- 最近输出

## 8. 风险和人工介入设计

风险不等于人工待办。系统应先判断是否能自愈。

分类：

- retryable：临时网络、锁、进程、超时。
- repairable：测试失败、发布漂移、状态漂移，可自动开修复任务。
- needs_review：质量、架构或设计风险，需要 reviewer。
- human_required：密钥、账号、外部授权、业务决策、发布窗口。

中台应把“系统正在修复”和“需要人工决策”分开展示。

## 9. 视觉稿入口

- 参考概念图：`docs/design/ops-workbench-concept.png`
- PC 本地视觉稿：`docs/design/ops-workbench-visual.html`
- 手机本地视觉稿：`docs/design/ops-workbench-mobile.html`

视觉稿包含：

- PC Web 工作台主屏。
- 手机工作台主屏。

PC 与手机视觉稿必须拆成两个独立入口。PC 文件只展示全屏工作台结果，不放手机预览、说明头或额外画布；手机文件只展示手机信息架构，避免两种尺寸互相影响。

## 10. P0 设计验收标准

- 中文设计稿清楚说明 PC Web 和手机端的不同信息架构。
- 视觉稿分别提供 PC 和手机尺寸入口。
- 视觉稿不得出现核心内容裁切、遮挡、文字压线或依赖横向滚动才能看到主流程的情况；宽表优先分页、折叠或纵向重组，不能把横向滚动作为默认布局方案。
- PC Web 单页工作台必须钉死为全屏应用壳：整体宽度和高度占满视口，不产生页面级横向或纵向滚动；需要滚动时限制在主内容区、侧栏或具体模块内部。手机尺寸按独立信息架构另行设计。
- 能力成熟度不能把 partial/scaffold 展示成已完成。
- 装饰性元素必须能解释其信息含义；没有业务含义的趋势线、斜线和填充图形不应出现在工作台视觉稿里。
- 视觉验收必须读取真实渲染结果：截图、DOM/layout 指标或浏览器可见状态至少覆盖 PC 首屏、PC 内部滚动后状态和手机首屏，不能只看 HTML/CSS 源码。
- 风险、自愈和人工决策在 UI 上分层。
- 后续实现任务能直接从本设计稿拆出后端能力矩阵和前端治理页。
